## Critical Issues

### C-1: `profiles` â€” Any authenticated user can self-escalate their role

**Policy:** `user can update own profile`

```sql
-- USING
(id = auth.uid())

-- WITH_CHECK
(id = auth.uid())
```

**Evidence (live DB query):**

The column-level `UPDATE` grant for the `authenticated` role on `profiles` covers **every column**, including `role`, `fixo_mensal`, `comissao_percent`, `ativo`, and `organization_id`. The policy only checks that the row being updated belongs to the authenticated user â€” it does not restrict which columns can be changed.

```
column_name               | privilege_type
--------------------------+---------------
role                      | UPDATE
fixo_mensal               | UPDATE
comissao_percent          | UPDATE
ativo                     | UPDATE
organization_id           | UPDATE
comissao_primeiro_mes_percent | UPDATE
```

**Risk:** Any authenticated user (e.g., a `designer` or `editor`) can issue:

```sql
UPDATE profiles
SET role = 'socio'
WHERE id = auth.uid();
```

After that update, `current_user_role()` returns `'socio'`, bypassing every other RLS policy in the system. This is a full privilege escalation vector.

**Reproducer:**

1. Authenticate as any non-admin user (e.g., `designer`).
2. `UPDATE profiles SET role = 'socio', fixo_mensal = 99999 WHERE id = auth.uid();`
3. All subsequent queries run with `current_user_role() = 'socio'`.

**Fix:**

```sql
-- Option A: Column-level REVOKE (preferred if the app only lets users change their own avatar/tema/contact info)
REVOKE UPDATE (role, fixo_mensal, comissao_percent, comissao_primeiro_mes_percent,
               ativo, organization_id, meta_prospects_mes, meta_fechamentos_mes,
               meta_receita_mes, data_admissao)
ON TABLE public.profiles
FROM authenticated;

-- Option B: Add a separate SECURITY DEFINER trigger that blocks role changes by non-admins
CREATE OR REPLACE FUNCTION public.prevent_self_role_change()
RETURNS trigger LANGUAGE plpgsql SECURITY DEFINER AS $$
BEGIN
  IF NEW.role <> OLD.role AND NOT (
    SELECT role FROM public.profiles WHERE id = auth.uid()
  ) = ANY(ARRAY['adm','socio']::user_role[]) THEN
    RAISE EXCEPTION 'Insufficient privilege to change role';
  END IF;
  RETURN NEW;
END;
$$;
```

### C-2: `profiles` â€” All active profiles expose salary and PII to every authenticated user

**Policy:** `team members can view active colleagues basic info`

```sql
-- USING
(ativo = true)
```

**Evidence:**

This policy matches for **every authenticated user** with no role restriction. There are no column-level SELECT restrictions. A `designer` or `videomaker` can therefore read:

- `fixo_mensal` (monthly salary)
- `comissao_percent` (commission percentage)
- `comissao_primeiro_mes_percent`
- `pix` (payment key â€” full financial PII)
- `endereco` (home address)
- `data_nascimento` (date of birth)
- `telefone`

for all active colleagues.

**Risk:** Any production-role employee (8 non-admin role types) can run:

```sql
SELECT nome, fixo_mensal, comissao_percent, pix, endereco
FROM profiles
WHERE ativo = true;
-- Returns salary + home address + PIX key for all 
```

**Reproducer:** Authenticate as `videomaker`, run query above. Returns full salary data for all active team members.

**Fix:**

```sql
-- Replace the broad policy with a restricted one that only exposes "safe" columns.
-- Use a view, or column-level REVOKE:

REVOKE SELECT (fixo_mensal, comissao_percent, comissao_primeiro_mes_percent,
               pix, endereco, data_nascimento, telefone,
               meta_prospects_mes, meta_fechamentos_mes, meta_receita_mes)
ON TABLE public.profiles
FROM authenticated;

-- Then grant those sensitive columns only to adm/socio (these are Postgres roles, 
-- so this needs to be done via service-role functions or a separate profile_sensitive view).

-- Alternatively: create a profiles_public view that exposes only safe columns,
-- point the app to that view for non-admin contexts.
```

## Important Issues

### I-1: `audit_log` â€” Any authenticated user can INSERT arbitrary records with any `ator_id`

**Policy:** `authenticated can insert audit log`

```sql
-- WITH_CHECK
true
```

**Risk:** The INSERT policy has `with_check = true` with no constraint on `ator_id`. A malicious or buggy client can:

1. Insert fake audit entries attributing actions to other users (`ator_id = <target_user_uuid>`).
2. Poison the audit trail (e.g., insert a fake "adm changed salary" record attributed to the socio).
3. Flood the table with garbage entries.

If the audit log is used for compliance or dispute resolution, this renders it worthless as evidence.

**Reproducer:**

```sql
INSERT INTO audit_log (entidade, entidade_id, acao, dados_antes, dados_depois, ator_id, justificativa)
VALUES ('profiles', '<victim-uuid>', 'UPDATE', '{}', '{"role":"adm"}', '<victim-uuid>', 'planted');
```

**Fix:**

```sql
-- Drop the permissive policy and replace with one that enforces ator_id = auth.uid()
DROP POLICY "authenticated can insert audit log" ON audit_log;

CREATE POLICY "authenticated can insert own audit log"
ON audit_log FOR INSERT TO authenticated
WITH CHECK (ator_id = auth.uid());

-- Ideally, audit_log should only be written by SECURITY DEFINER triggers,
-- not directly by clients. Revoke client INSERT entirely:
-- REVOKE INSERT ON audit_log FROM authenticated;
```

### I-2: `leads` â€” All authenticated users can SELECT all lead data (no role restriction)

**Policy:** `authenticated read leads`

```sql
-- USING
true
```

**Risk:** The spec states "comercial team sees leads." However, the SELECT policy is `true` â€” meaning any `assessor`, `designer`, `editor`, `videomaker`, or `coordenador` can read all leads including:

- `valor_proposto` (proposed contract value)
- `contato_principal`, `email`, `telefone` (prospect PII)
- `info_briefing` (strategic briefing notes)
- `stage`, `motivo_perdido` (sales pipeline intelligence)

This is a data classification concern: production roles have no business need for prospecting pipeline data.

**Reproducer:** Authenticate as `designer`. `SELECT * FROM leads;` returns all records.

**Fix:**

```sql
DROP POLICY "authenticated read leads" ON leads;

CREATE POLICY "comercial/adm/socio/coord read leads"
ON leads FOR SELECT TO authenticated
USING (
  current_user_role() = ANY(ARRAY['adm','socio','comercial','coordenador']::user_role[])
);
```

### I-3: `leads` â€” `coord update leads in marco_zero` has a USING/WITH_CHECK mismatch

**Policy:** `coord update leads in marco_zero`

```sql
-- USING (row must be in this state to be targeted)
((current_user_role() = 'coordenador'::user_role) AND (stage = 'marco_zero'::lead_stage))

-- WITH_CHECK (row must satisfy this after the update)
((current_user_role() = 'coordenador'::user_role) AND (stage = ANY (ARRAY['marco_zero'::lead_stage, 'ativo'::lead_stage])))
```

**Risk:** A coordinator can update a `marco_zero` lead and set `stage = 'ativo'`, transitioning it to active status. The USING clause prevents them from targeting `ativo` rows, but the WITH_CHECK allows writing `ativo` as the resulting state. Additionally, there is no column restriction: a coordinator updating a `marco_zero` lead can also change `comercial_id`, `valor_proposto`, `assessor_alocado_id`, etc., which are fields that should only be managed by comercial or adm/socio.

**Fix:**

If the intent is to let coordinators confirm/graduate marco_zero to ativo, the policy is functional but should be explicitly documented. If it is NOT intentional:

```sql
DROP POLICY "coord update leads in marco_zero" ON leads;

CREATE POLICY "coord update leads in marco_zero"
ON leads FOR UPDATE TO authenticated
USING (
  current_user_role() = 'coordenador'
  AND stage = 'marco_zero'
)
WITH CHECK (
  current_user_role() = 'coordenador'
  AND stage = 'marco_zero'  -- disallow stage transitions
);
```

If coordinators legitimately need to promote to `ativo`, lock down which columns they can touch via column-level grants.

Follows from C-1 (the same `user can update own profile` policy). Beyond role escalation, a user can:

- Set `ativo = false` on their own account (soft-delete/deactivate self, potentially breaking RLS checks elsewhere that filter `ativo = true`).
- Set `organization_id` to another organization's UUID (cross-tenant data contamination if multi-org is in use).

**Fix:** Included in the C-1 fix (revoke column-level UPDATE for these columns from `authenticated`).

### I-5: `storage.objects` â€” avatar UPDATE policy has no `with_check`

**Policy:** `users update own avatar`

```sql
-- USING
((bucket_id = 'avatars'::text) AND ((storage.foldername(name))[1] = (auth.uid())::text))

-- WITH_CHECK
null  -- <-- absent
```

**Risk:** The USING clause correctly restricts which objects a user can target (only their own folder). However, without a matching `WITH_CHECK`, after the storage engine applies the update the resulting row is not re-validated. In Supabase Storage, `UPDATE` on `storage.objects` is used when replacing a file. Without `with_check`, a user could potentially rename/move the object to a different path after the update is applied (e.g., rename their avatar to another user's path), overwriting another user's avatar.

**Fix:**

```sql
DROP POLICY "users update own avatar" ON storage.objects;

CREATE POLICY "users update own avatar"
ON storage.objects FOR UPDATE TO authenticated
USING   ((bucket_id = 'avatars') AND ((storage.foldername(name))[1] = auth.uid()::text))
WITH CHECK ((bucket_id = 'avatars') AND ((storage.foldername(name))[1] = auth.uid()::text));
```

Same fix applies to `adm/socio update any avatar` which also has `with_check = null`.

## Minor / Observations

### M-1: `cron_runs` â€” RLS enabled but no policies defined

RLS is enabled on `cron_runs` (`relrowsecurity = true`) but there are **zero policies** in `pg_policies` for this table. In Postgres, RLS with no policies defaults to **deny all** for non-superusers. This is likely intentional (cron jobs run as service role which bypasses RLS), but worth confirming:

- If any client-side code tries to read `cron_runs`, it will receive an empty result set with no error, which could be confusing.
- If the intent is "adm/socio can monitor cron job execution", a SELECT policy should be added.

**Recommendation:** Add an explicit SELECT policy for adm/socio, or document that this table is service-roleâ€“only.

---

### M-2: `checklist_step` and `client_monthly_checklist` UPDATE â€” missing `with_check`

Both UPDATE policies have `with_check = null`. While their USING clauses correctly scope which rows can be targeted, the absence of `with_check` means no validation is applied to the resulting row post-update. A designer assigned to client A could update a checklist row's `client_id` (if that column is updatable) to point it at client B.

**Verification:** Confirm whether `client_id` / `checklist_id` are protected by column-level grants or application logic. If not, add matching `with_check` clauses.

## Appendix: Policy inventory

All 26 public tables have `relrowsecurity = true`. No table has `relforcerowsecurity = true` (this only matters for table owners, i.e., Postgres superusers running as the table owner â€” not a concern for the `authenticated` role).

`current_user_role()` is `SECURITY DEFINER` (`prosecdef = true`) and `STABLE` (`provolatile = 's'`). It reads from `public.profiles` where `id = auth.uid()`. If a user's profile row is missing (e.g., during registration), the function returns `NULL`, causing all role-based `ANY(ARRAY[...])` checks to fail â€” effectively locking out the user. This is safe-by-default behavior.
